Speculative Decoding Summary
============================

Workload
- GPU: NVIDIA A100-SXM4-40GB on GCP
- Target model: Qwen/Qwen3-8B
- Main prompt length: 4096 tokens
- Main generation length: 128 tokens
- Batch size: 1
- Dtype: bf16

What I implemented
- Added a resumable HF speculative sweep runner:
  - run_speculative_qwen3_sweep.py
- Added a sweep summarizer:
  - summarize_speculative_qwen3_sweep.py
- Added a comparison plotting script:
  - plot_speculative_vs_baseline.py
- Extended hf_a100_benchmark.py in two directions:
  - HF speculative decoding path with NVTX ranges for profiling
  - Native TensorRT-LLM draft-target speculative decoding path

What I tested
1. HF speculative decoding sweep
- Draft models tested:
  - Qwen/Qwen3-0.6B
  - Qwen/Qwen3-1.7B
  - Qwen/Qwen3-4B
- Draft lengths tested:
  - k = 1, 2, 4, 8, 16

2. HF speculative profiling
- Profiled the best HF speculative point from the sweep:
  - target = Qwen/Qwen3-8B
  - draft = Qwen/Qwen3-1.7B
  - k = 4
- Added NVTX ranges around:
  - spec_target_prefill
  - spec_draft_prefill
  - spec_draft_propose
  - spec_target_verify
  - spec_target_corrective
  - spec_draft_corrective

3. Native TensorRT-LLM speculative decoding
- Used TensorRT-LLM PyTorch backend native draft-target decoding
- Draft model used:
  - Qwen/Qwen3-1.7B
- On this VM, TRT-LLM draft-target required the draft model as a local HF snapshot path, not just the repo ID string

Main results
------------

HF baseline vs HF speculative
- HF baseline:
  - total_generate_ms = 7285.69
  - end_to_end_tokens_per_sec = 17.57
- Best HF speculative point:
  - draft = Qwen/Qwen3-1.7B
  - k = 4
  - total_generate_ms = 10733.26
  - end_to_end_tokens_per_sec = 11.93

Conclusion for HF path
- HF speculative decoding was slower than plain HF on this workload.
- The best HF speculative point still achieved only about 0.68x the HF baseline throughput.

Why HF speculative was slow
- The HF implementation used loose Python/HF orchestration.
- Draft generation was the dominant cost, not verification.
- Main issues observed:
  - many small kernel launches
  - many copy/concat operations
  - Python-side control flow and synchronization
  - repeated tiny draft steps

HF profiling findings
---------------------

Profiled config
- target = Qwen/Qwen3-8B
- draft = Qwen/Qwen3-1.7B
- k = 4
- prompt_len = 4096
- max_new_tokens = 128

Stage-level GPU projected time from nsys
- spec_draft_propose: ~9969 ms
- spec_target_verify: ~3897 ms
- spec_target_prefill: ~870 ms
- spec_target_corrective: ~830 ms
- spec_draft_corrective: ~629 ms
- spec_draft_prefill: ~123 ms

High-level conclusion
- The draft proposal stage was the biggest bottleneck.
- This matched the design concern that a loose HF orchestration path is not a serious speculative benchmarking path.

CUDA API pressure comparison
----------------------------

HF speculative profile
- cudaLaunchKernel calls: 388,507
- cudaMemcpyAsync calls: 1,151
- cudaStreamSynchronize calls: 1,150

Native TRT-LLM speculative profile
- cudaLaunchKernel calls: 8,562
- cudaLaunchKernelExC_v11060 calls: 3,579
- cudaMemcpyAsync calls: 2,094
- cudaStreamSynchronize calls: 1,009
- cudaGraphLaunch_v10000 calls: 66

Interpretation
- The serious-path improvement is mostly about drastically reducing launch count and moving execution into a much tighter runtime.
- Even though memcpy counts are not lower in the whole-process API summary, the launch count collapse is substantial and the end-to-end latency improvement confirms that the native path is much tighter.

TensorRT-LLM results
--------------------

Plain TensorRT-LLM baseline
- total_generate_ms = 2046.32
- prefill_ms = 283.55
- decode_ms = 1755.18
- end_to_end_tokens_per_sec = 62.55

Native TensorRT-LLM speculative decoding
- draft = Qwen/Qwen3-1.7B
- max_draft_len = 4
- total_generate_ms = 1591.75
- prefill_ms = 429.94
- decode_ms = 1154.33
- end_to_end_tokens_per_sec = 80.41

Conclusion for serious benchmark path
- Native TensorRT-LLM speculative decoding beat plain TensorRT-LLM baseline.
- Relative improvement:
  - latency: 2046.32 ms -> 1591.75 ms
  - throughput: 62.55 tok/s -> 80.41 tok/s
- This is about a 1.29x throughput improvement for the tested setting.

Recommended path forward
------------------------
- Do not use the loose HF orchestration path as the main serious speculative benchmark path.
- Use native TensorRT-LLM draft-target speculative decoding for serious measurement.
- If deeper kernel analysis is needed:
  - use the native TRT-LLM path as the profiling target
  - install a newer Nsight Compute on the VM, because the current ncu version on the instance is too old for the installed driver/runtime
